\begin{abstract}
Menu Intelligence is an end-to-end service for converting pasted menu text,
PDF menus, and menu images into a structured JSON menu representation. The
project is framed as an NLP pipeline with three main layers: document text
extraction, item-level category classification, and deterministic structured
field extraction for names, prices, and sizes. The final course dataset is
\texttt{items.v2.jsonl}, a derived item-level dataset with
432 annotated menu items from 12 public restaurant or hotel dining sources,
using source-level train/validation/test splits. On the held-out
\texttt{items.v2} test split, the strongest model is a TF-IDF word/character
n-gram feature union with Logistic Regression, which reaches 0.7361 accuracy
and 0.7254 Macro-F1. This classifier is the current backend category model,
while menu item field extraction remains deterministic. The repository is
available at
\url{https://github.com/philippovdev/nlp-menu-intelligence}.
\end{abstract}

\section{Introduction}

Restaurant menus are still frequently published as free-form text, PDFs, or
images intended for human reading rather than structured reuse. This creates
friction for digital menu management, catalog maintenance, search, delivery
integration, and downstream analytics. The practical goal of this project is
to build a web service that accepts raw menu content and returns structured
menu items that can be reviewed and edited in a browser.

From an NLP perspective, the task is not a single model problem. It is a
pipeline that begins with raw document text, continues through line splitting
and menu-item detection, predicts a fixed category label for each candidate
item, and extracts structured fields such as item name, price, and size. This
report therefore treats the system as a composition of measurable tasks rather
than as a monolithic application.

\paragraph{Contributions.}
This project makes four concrete contributions. First, it implements a working
menu-structuring service with a deployed frontend and FastAPI backend. Second,
it defines a stable item-level JSONL schema and two public dataset slices,
culminating in the training-ready \texttt{items.v2.jsonl} release. Third, it
measures heuristic, classical, and transformer classification baselines on a
fixed split. Fourth, it integrates the best measured classifier into the real
backend parsing pipeline while preserving deterministic field extraction and
the existing API contract.

\subsection{Team}

This is a solo project. All work was completed by one contributor: problem
definition, dataset curation, annotation design, baseline experiments, model
training, backend and frontend implementation, deployment, and report
preparation.

\section{Related Work}

The closest published work to Menu Intelligence does not come from one single
benchmark. Instead, the relevant literature splits across three neighboring
areas: short text and product-title categorization, food-domain language
understanding, and structured extraction from semi-structured restaurant or
document text.

Short text and product categorization work motivates the classical baselines
used in this project. fastText shows that linear text models remain strong and
efficient on short classification inputs \cite{joulin-etal-2017-bag}. Product
title categorization papers report high accuracy on fixed taxonomies and
provide the closest mature analog to menu item classification, even though
their domains are e-commerce rather than restaurant menus
\cite{xia-etal-2017-large,cevahir-murakami-2016-large}.

Food-domain NLP is more fragmented. Prior work includes food-item healthiness
classification \cite{wiegand-klakow-2013-towards}, recipe-to-dish-name
recognition \cite{hu-etal-2023-diner}, and large-scale menu normalization or
clustering \cite{latif-etal-2025-restaurant}. These papers confirm that food
text has domain-specific lexical and compositional structure, but they do not
define the same supervised fixed-label classification task used here.

Structured extraction evidence is stronger in adjacent domains than in menus
themselves. Restaurant-order extraction from dialogue
\cite{peng-etal-2017-may} and receipt or form understanding resources such as
FUNSD, CORD, and SROIE \cite{jaume2019funsd,park2019cord,huang2019icdar2019}
are useful analogs for recovering prices and other fields from noisy,
semi-structured text. However, they still do not match the full task shape of
this project: short menu item text, fixed category labels, deterministic slot
extraction, and upstream OCR support in one public benchmark.

\section{Model Description}

The implemented system is a hybrid pipeline rather than an end-to-end language
model. Raw input can arrive as pasted text, embedded-text PDF, or image. PDF
files are processed through embedded text extraction first; images are passed
through OCR. The extracted text is then normalized and split into candidate
lines. Each line is classified as a category header, a menu item, or noise.
Menu-item lines are assigned a category label, and deterministic parsing
extracts the item name, price list, and size list before the service returns a
structured JSON response for review.

The deployed classification stage is now model-backed. The backend ships a
scikit-learn TF-IDF word/character feature union with Logistic Regression
trained on \texttt{items.v2.jsonl}, loads it at application startup, and uses
it as the default item-level category predictor. Deterministic extraction
remains rule-based. If the classifier artifact is unavailable or model
confidence is too weak, the parser falls back to conservative heuristic
categorization. This design keeps category prediction learned, field
extraction deterministic, and runtime behavior explicit. ONNX remains a future
export target rather than the current shipped runtime.

\section{Dataset}

The main course dataset is an item-level JSONL corpus built for menu
classification and deterministic slot evaluation. Each record stores a stable
item identifier, source identifier, restaurant identifier, split assignment,
language tag, raw item text, one category label, and structured slots for
name, prices, and sizes. The active label set contains 12 classes:
\texttt{salads}, \texttt{soups}, \texttt{mains}, \texttt{pizza},
\texttt{pasta}, \texttt{burgers}, \texttt{sides}, \texttt{desserts},
\texttt{breakfast}, \texttt{drinks\_hot}, \texttt{drinks\_cold}, and
\texttt{other}.

Two dataset slices are present in the repository. \texttt{items.v1.jsonl} is a
small first experimental subset used for early classical baselines.
\texttt{items.v2.jsonl} is the training-ready release used for the final
classification comparison and the shipped backend model. The paired manifest
stores public source provenance, collection date, source type, license note,
and notes, while raw captures remain outside git.

The report uses \texttt{items.v2} as the fixed evaluation dataset. It contains
432 annotated menu items from 12 public restaurant or hotel dining sources,
split by source and restaurant to avoid leakage between similar items. The
release is intentionally a source-grounded synthetic expansion rather than a
source-faithful raw crawl: public menus are used as provenance anchors and
lexical references, but the released item texts are normalized into consistent
English forms with normalized \texttt{RUB} prices. This keeps the public
repository lightweight and reproducible, but it also limits how directly the
dataset reflects raw restaurant menu wording.

\section{Experiments}

\subsection{Metrics}

The main classification metrics are accuracy and Macro-F1. Macro-F1 is the
primary model-selection metric because the project treats all 12 categories as
equally important. For the heuristic end-to-end pipeline, the report also
tracks exact-match accuracy for price and size extraction on normalized
item-level gold annotations.

\subsection{Setup}

All trained classifiers use the fixed \texttt{items.v2} source-level split:
288 training items, 72 validation items, and 72 held-out test items. The
classical baselines use raw item text as input and predict one category label
per item. The first transformer benchmark fine-tunes
\texttt{distilbert-base-uncased} as a sequence classifier with maximum
sequence length 64, batch size 16 for training, batch size 32 for evaluation,
learning rate $2 \times 10^{-5}$, weight decay 0.01, warmup ratio 0.1, and
early stopping on validation Macro-F1.

The product system itself is broader than the classification experiments: it
also supports PDF and image intake, embedded PDF text extraction, OCR for
images, deterministic line segmentation, and deterministic slot extraction.
However, OCR quality is not the main evaluated contribution of this report.
The quantitative comparison therefore focuses on item-level category
classification and reports deterministic extraction only for the heuristic
pipeline.

To keep end-to-end product quality visible, the repository also includes a
small real-world evaluation slice with 12 source-grounded examples: four
pasted-text cases, four embedded-text PDF fixtures, and four rendered image
fixtures. On this slice, the current deployed backend recovers text, category,
price, and size exactly on every example. This set is useful as a lightweight
regression check, but it is easier than arbitrary real phone photos and is not
used as the main benchmark.

\subsection{Methods}

Seven methods are compared:
\begin{itemize}
  \item heuristic end-to-end parser with keyword category rules,
  \item TF-IDF + Logistic Regression,
  \item TF-IDF + Linear SVM,
  \item TF-IDF word/char union + Logistic Regression,
  \item TF-IDF + sigmoid-calibrated Linear SVM,
  \item DistilBERT sequence classifier,
  \item XLM-RoBERTa sequence classifier.
\end{itemize}

The heuristic system is useful as a product baseline because it runs through
the real parser and produces both category and slot outputs. The two TF-IDF
baselines provide direct classical comparisons on the fixed split. The
DistilBERT and XLM-RoBERTa runs are included as practical transformer
benchmarks on the same dataset and split.

\section{Results}

Table~\ref{tab:main-results} summarizes the measured results already saved in
the repository artifacts. The strongest held-out test result on
\texttt{items.v2} is TF-IDF word/char union + Logistic Regression with 0.7361
accuracy and 0.7254 Macro-F1. Plain TF-IDF + Logistic Regression reaches
0.7222 accuracy and 0.7009 Macro-F1, while TF-IDF + Linear SVM reaches 0.7083
accuracy and 0.6930 Macro-F1 on the same test split. The probability-aware
calibrated Linear SVM remains below the union model, and the two tested
transformer baselines also underperform the strongest sparse feature pipeline.

The per-class F1 scores show that the task is not uniformly difficult. The
shipped TF-IDF word/char union + Logistic Regression model reaches perfect
test F1 on \texttt{burgers} and \texttt{pizza}, and strong scores on
\texttt{desserts}, \texttt{drinks\_cold}, and \texttt{salads} (all 0.9091).
The hardest classes are \texttt{sides} (0.0), \texttt{mains} (0.3636), and
\texttt{other} (0.5263). The dominant confusion is \texttt{sides}
$\rightarrow$ \texttt{other}, with smaller overlaps between
\texttt{drinks\_cold} and \texttt{drinks\_hot}, and between broad savory
categories such as \texttt{mains}, \texttt{soups}, and \texttt{breakfast}.
This pattern suggests that short menu texts with strong lexical anchors are
easier than context-dependent categories whose boundaries overlap with other
savory dishes.

The transformer results do not support switching away from classical methods
yet. DistilBERT reaches 0.5556 test accuracy and 0.5620 test Macro-F1, while
the multilingual XLM-RoBERTa follow-up reaches the same 0.5556 test accuracy
but only 0.5462 test Macro-F1 on the same fixed split. On this dataset size
and label taxonomy, the measured evidence still favors sparse lexical features
over the tested transformer configurations. The union model likely benefits
from the combination of short menu texts, strong lexical anchors, and
character n-grams that help absorb noisy spelling, morphology, transliteration,
and OCR-like variation.

\subsection{Limitations}

The report has four important limitations. First, \texttt{items.v2} is a
source-grounded synthetic expansion rather than a source-faithful raw crawl, so
the released public text is cleaner and more regular than uncontrolled menu
captures. Second, the heuristic extraction metrics are measured on normalized
item-level annotations and should not be read as a direct OCR robustness
result. Third, the first transformer run is only one practical benchmark, not
an exhaustive transformer search. Fourth, the current production parser is a
hybrid system: category assignment is learned, but name, price, and size
extraction remain deterministic by design.

\section{Conclusion}

Menu Intelligence delivers a real menu-structuring service and a measurable NLP
pipeline behind it. The project now has a stable item-level schema, public
dataset releases, a fixed training-ready classification dataset, measured
heuristic and learned baselines, and a backend that uses the best measured
classifier in production. The strongest held-out result on \texttt{items.v2}
is TF-IDF word/char union + Logistic Regression at 0.7361 accuracy and 0.7254
Macro-F1, which outperforms the plain TF-IDF baselines, the calibrated Linear
SVM comparison, and the tested transformer runs.

The main remaining work is not to redesign the service but to deepen the
evaluation. Future steps should focus on richer error analysis, better OCR
stress tests, possible transformer follow-up experiments only if they improve
over the classical baseline, and ONNX export only after it can preserve the
current sparse-model quality. The core report claim is therefore modest but
well supported: for this dataset and taxonomy, a hybrid parser with OCR or PDF
ingestion upstream, a sparse learned category model, and deterministic field
extraction is currently the most practical solution.
